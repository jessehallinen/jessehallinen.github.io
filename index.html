SURVIVAL FPS GAME WITH MAGIC

A first-person shooter survival game with magic. The goal of this project was to make a local multiplayer split-screen game with as many AI enemies as possible.

Development problems and solutions

The very annoying problem with Unity is that it doesn't really support split-screen games. Unity doesn't allow multiple audio listeners or event systems, so the only option is to write some scripts yourself but it is hard to get them working properly in every situation. Audio listener problem can be solved by keeping the audio listener in same position and moving audio sources around it in relation to the every player position. The event system problem can be solved by modifying the GUISystem files in UnityExtensions folder to allow multiple eventsystems or by writing some scripts on top of the eventsystem. Both of the solutions are cumbersome.

The other obvious problem was how to get hundreds of enemies and still get high framerate. There are many small things to do to improve fps but one of the biggest things was to find out how to make the enemies be as lightweight as possible. After testing every combination this is what I found out:

  • An enemy model made from boxes vs a few hundred polygon model. A noticeable fps difference so if only want as many enemies as possible then use the box model.
  • Enemy ragdolls. These are heavy so better not to use
  • Accurate hit colliders for every limb. Kind of heavy so better not to use unless really want to be able to shoot enemies in the foot or arm.
  • NavMeshAgent vs custom solution. NavMeshAgents are actually pretty fast so they can be used.
  • Enemy rigidbodies. Somewhat heavy so better not to use.
  • Rigging and animations in Blender vs Unity animations with no rigging which requires separate game objects for every limb. Using Blender for rigging and animations was the more lightweight solution.
  • Using only the minimum amount of bones in enemy skeletons gave some extra fps.
  • Removing all the unnecessary child transforms from the enemy game objects, like IK bones. This gave a few more fps.

Castles in War

A strategy game where a player needs to conquer the enemy castles. The castles have multiple different upgrade options and there are also gold mines which generate gold faster than castles. The game is designed for mobile devices and was published in App Store and Google Play, but currently it's only available in Google Play.

Development problems and solutions

The knights couldn't be individual game objects because performance was too bad. The solution was to parent every knight in a same queue under an other game object and then move that queue object back and forth to create an illusion of the knights continuously moving forward. This created a new problem: cutting knight queue at the correct position. This was solved by calculating how many knights fit between the cut position and the end of the queue object.

King's Castle TD

A tower defence game where the player needs to protect the king from the monsters. The twist here is that the player can build walls to change the route the monsters can take. The towers can be build on top of the walls.

Development problems and solutions

There could be so many monsters at the same time that towers couldn't just loop through every monster and calculate which one was closest to the king. This was solved by splitting the map area into multiple squares called buckets. Each bucket hold the list of the monsters which currently were in the bucket's area. Then each tower needed only to check the buckets closest to it.

World Conquering Game

A turn-based strategy game where the player needs to conquer the whole world. Kind of like Risk board game. The player can choose any nation as a starting point. All countries are divided into smaller regions and each nation gets reinforcement units at the start of their turn based on the number of regions the nation holds. Before battle the player positions his troops and then in the actual battle the player has no control and the troops just charge against the enemy.

Development problems and solutions

The biggest problem was creating an accurate world map where countries were divided into smaller regions. The solution required multiple tools. First I downloaded geographical data from naturalearthdata.com and a list of the world flags from somewhere (Couldn't find the site anymore). Then I run the geographical world data through the mapshaper.org to get more simplified version of the map. Then I wrote a dozen of python scripts for QGIS (a geographic information system application for editing geospatial data) to split the map into smaller regions and to convert the world map data into a format I could use in Unity to generate the world map.
